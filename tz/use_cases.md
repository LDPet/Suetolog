# Use Cases - «Голосовой напоминальщик»

Единый документ пользовательских сценариев Telegram-бота. Он связывает требования из `tz/tz.md` с актуальной архитектурой, голосовым пайплайном и пайплайнами рассылок.

**Источники scope:** `tz/tz.md`, `tz/ARCHITECTURE.md`, `tz/VOICE_PIPELINES.md`, `tz/MAILING_PIPELINES.md`, `tz/tables.md`.

## Границы продукта

Проект - Telegram-бот на `aiogram` и Django backend. Основной интерфейс пользователя - Telegram. Пользователь идентифицируется по `chat_id`; отдельная регистрация, сайт, личный кабинет и роли пользователей не входят в MVP.

MVP использует:

- PostgreSQL как источник истины для пользователей, задач, напоминаний, реакций, событий и технических статусов доставки;
- Redis только как Celery broker;
- Celery Beat как единственный планировщик периодических jobs;
- Celery Worker для фоновых рассылок и напоминаний;
- Yandex SpeechKit для синхронного STT коротких Telegram voice;
- YandexGPT / Foundation Models для разбора русского текста в структуру задачи;
- единый часовой пояс `DEFAULT_TIMEZONE` из `config/settings.py` (по умолчанию `Europe/Moscow`).

Главное архитектурное правило: Telegram handlers, Celery tasks и внешние API-клиенты не содержат бизнес-логику. Они принимают событие, вызывают сервис и возвращают результат.

## Scope MVP

Ключевой цикл продукта:

1. Пользователь создаёт задачу голосом или текстом.
2. Voice проходит Telegram File API, проверки лимитов, Yandex SpeechKit STT и YandexGPT parser.
3. Text сразу передаётся в YandexGPT parser.
4. `TaskCreationService` создаёт `Task`, опциональный `Reminder` и `TaskEvent` в одной транзакции.
5. Каждое утро в `MORNING_DIGEST_HOUR`:00 MSK (по умолчанию 09:00) бот присылает дайджест задач на день.
6. В назначенное время бот отправляет напоминание по записи `Reminder`.
7. Реакция ✅ делает задачу выполненной, реакция ❌ отменяет задачу.
8. Если реакции нет, в `EVENING_MISSED_CHECK_HOUR`:00 MSK (по умолчанию 20:00) бот спрашивает новую дату для переноса.
9. Команда «без даты» показывает задачи без `due_to`; пользователь может назначить дату или удалить задачу.

Follow-up: полный recurrence-cycle для повторяющихся задач, недельная статистика, расширенная пагинация и промышленные delivery-статусы/метрики. При этом поля повторения и базовая страничность списков полезно заложить уже в MVP.

## Акторы

| Актор | Роль в сценариях |
| --- | --- |
| Пользователь Telegram | Создаёт задачи, смотрит списки, отмечает выполнение, отменяет и переносит задачи |
| Telegram Bot на aiogram | Принимает команды, voice, text, callback-кнопки и реакции; отправляет ответы |
| Django backend | Хранит модели, миграции, сервисы и бизнес-логику |
| PostgreSQL | Постоянно хранит пользователей, задачи, напоминания, события, реакции и технические статусы |
| Redis broker | Передаёт Celery jobs от Beat к Worker, но не хранит бизнес-состояние |
| Celery Beat | Публикует jobs для дайджеста, due reminders и вечерней проверки |
| Celery Worker | Исполняет фоновые jobs и вызывает сервисы рассылок |
| Yandex SpeechKit | Преобразует короткое OGG Opus voice в русский текст |
| YandexGPT parser | Нормализует русский текст в JSON задачи |

## Компоненты и сервисы

| Компонент | Ответственность |
| --- | --- |
| `UserService` | Найти или создать пользователя по `chat_id` |
| `VoiceService` / audio pipeline | Скачать voice, проверить длительность/размер/формат, вызвать SpeechKit, удалить временный файл |
| `ParserService` | Вызвать YandexGPT, валидировать JSON Schema, вернуть `ParsedTaskInput` |
| `TaskCreationService` | Создать `Task`, `Reminder` и `TaskEvent` в одной транзакции |
| `TaskService` | Списки, назначение даты, удаление, выполнение, отмена, перенос, события |
| `ReminderService` | Создать reminder, выбрать due reminders, сохранить `sent_time` и `message_id`, найти reminder по реакции |
| `MailingService` | Подготовить утренний дайджест и вечерний вопрос о переносе |
| `TelegramSender` | Отправить сообщения, списки, напоминания и inline-кнопки |

## Основные состояния

### Task

- `active` - задача актуальна и попадает в списки/напоминания.
- `done` - пользователь поставил ✅.
- `cancelled` - пользователь поставил ❌.
- `deleted` - пользователь удалил задачу через inline-кнопку.

### Reminder

Минимальные поля:

- `reminder_time` - когда нужно отправить напоминание;
- `sent_time` - когда отправка успешно зафиксирована;
- `reaction` - `done`, `cancelled` или пусто;
- `message_id` - Telegram message id для связи реакции с задачей;
- `task_id` - связанная задача.

Логические статусы можно выводить из полей или добавить технические поля: `pending`, `due`, `sent`, `failed`, `cancelled`. Redis не должен быть единственным местом хранения delivery-состояния.

### TaskEvent

События нужны для истории, отладки дублей и follow-up статистики:

- `created`;
- `reminder_sent`;
- `completed`;
- `cancelled`;
- `rescheduled`;
- `date_set`;
- `deleted`;
- `digest_sent` - опционально.

## MVP Use Cases

### UC-01. Первое обращение пользователя

**Цель:** привязать все дальнейшие действия к Telegram `chat_id`.

**Предусловия:**

- пользователь открыл бота в Telegram;
- записи `User` с таким `chat_id` может ещё не быть.

**Основной сценарий:**

1. Пользователь отправляет `/start`, voice, text или поддерживаемую команду.
2. Handler получает `chat_id`.
3. `UserService` ищет пользователя по `chat_id`.
4. Если пользователя нет, сервис создаёт `User`.
5. Бот отвечает приветствием или продолжает обработку исходного действия.

**Постусловия:**

- в PostgreSQL есть ровно один `User` для данного `chat_id`;
- дальнейшие задачи и напоминания создаются только для этого пользователя.

**Исключения:**

- Повторное обращение существующего пользователя не создаёт дубль.
- Неподдерживаемое сообщение получает понятную подсказку.

**Связанные компоненты:** `StartHandler`, `UserService`, PostgreSQL.

---

### UC-02. Создание задачи голосом

**Цель:** пользователь быстро создаёт задачу голосовым сообщением.

**Предусловия:**

- пользователь пишет боту в Telegram;
- voice не превышает `VOICE_MAX_DURATION_SEC` и `VOICE_MAX_SIZE_BYTES` из `config/settings.py`;
- Yandex SpeechKit и YandexGPT настроены в `config/settings.py`;
- пользователь произносит одну задачу свободным русским текстом.

**Основной сценарий:**

1. Пользователь отправляет Telegram voice.
2. Handler создаёт/находит пользователя по `chat_id`.
3. Бот отправляет короткий статус обработки.
4. Audio pipeline скачивает OGG Opus через Telegram File API.
5. Система проверяет длительность, размер и формат.
6. `VoiceService` вызывает синхронный Yandex SpeechKit STT.
7. `ParserService` передаёт transcript в YandexGPT с текущим `now`, timezone и JSON Schema.
8. Parser возвращает валидный `ParsedTaskInput`.
9. `TaskCreationService` создаёт `Task(active)`, опциональный `Reminder` и `TaskEvent(created)` в одной транзакции.
10. Временный аудиофайл удаляется.
11. Бот подтверждает создание задачи и показывает распознанные данные.

**Постусловия:**

- задача принадлежит текущему `User`;
- `Task.status = active`;
- `Task.raw_text` или эквивалентный audit-контекст не содержит секретов;
- `Reminder` создан только если есть будущие дата и точное время;
- частично созданные данные не остаются в БД при ошибке.

**Исключения:**

- Voice слишком длинный или большой: SpeechKit и GPT не вызываются, задача не создаётся.
- Telegram File API недоступен: задача не создаётся, пользователь получает ошибку получения аудио.
- SpeechKit вернул пустой transcript: бот просит повторить голосом или отправить текстом.
- YandexGPT вернул невалидный JSON или пустой `title`: задача не создаётся.
- `due_to` в прошлом: задача не создаётся, бот просит будущую дату.
- Yandex 401/403: не ретраить, залогировать техническую ошибку без ключей.
- Timeout/5xx SpeechKit или GPT: допустим один retry, затем понятная временная ошибка.

**Бизнес-правила:**

- Voice pipeline в MVP синхронный; Celery не используется для AI-разбора.
- Голосовой и текстовый ввод сходятся в один `TaskCreationService`.
- Parser возвращает только JSON без Markdown и не придумывает отсутствующие факты.
- `repeat_type` может быть `daily`, `weekly`, `monthly` или `null`; полный цикл повторений см. UC-F01.

**Связанные компоненты/пайплайны:** `VoiceHandler`, Telegram File API, `VoiceService`, Yandex SpeechKit, `ParserService`, YandexGPT, `TaskCreationService`, PostgreSQL.

### UC-03. Создание задачи текстом

**Цель:** пользователь создаёт задачу без голосового сообщения.

**Предусловия:**

- пользователь отправляет обычный текст боту;
- текст не является служебной командой: `/start`, `/undated`, «без даты», callback-ответом на перенос и т.п.;
- YandexGPT parser доступен.

**Основной сценарий:**

1. Пользователь отправляет текст с задачей.
2. Handler создаёт/находит пользователя по `chat_id`.
3. `ParserService` передаёт текст в YandexGPT с тем же контрактом, что и transcript voice.
4. Parser возвращает `ParsedTaskInput`.
5. `TaskCreationService` создаёт `Task`, опциональный `Reminder` и `TaskEvent(created)`.
6. Бот подтверждает создание задачи.

**Постусловия:**

- задача создана тем же способом, что и после voice transcript;
- STT и Telegram File API не вызываются.

**Исключения:**

- Текст нельзя интерпретировать как задачу: бот просит сформулировать проще.
- Parser недоступен или вернул невалидный JSON: задача не создаётся.
- Дата в прошлом: задача не создаётся.

**Бизнес-правила:**

- Текстовый ввод входит в MVP по актуальному voice/text pipeline.
- Отдельной бизнес-логики создания для text быть не должно.

**Связанные компоненты/пайплайны:** `TextHandler`, `ParserService`, YandexGPT, `TaskCreationService`.

---

### UC-04. Задача без даты

**Цель:** сохранить задачу, если пользователь не знает дату или явно сказал «без даты».

**Предусловия:**

- пользователь создаёт задачу голосом или текстом;
- parser вернул `due_to = null` или пользователь явно указал «без даты».

**Основной сценарий:**

1. `TaskCreationService` получает `ParsedTaskInput` без `due_to`.
2. Сервис создаёт `Task(active)` с `due_to = null`.
3. `Reminder` не создаётся.
4. Система создаёт `TaskEvent(created)`.
5. Бот подтверждает создание задачи без даты.

**Постусловия:**

- задача попадает в список «без даты»;
- задача не попадает в due reminders до назначения даты.

**Исключения:**

- Если title пустой, задача не создаётся.

**Связанные компоненты:** `TaskCreationService`, `TaskService`, PostgreSQL.

---

### UC-05. Просмотр задач без даты

**Цель:** пользователь видит задачи, для которых ещё не назначены дата и время.

**Предусловия:**

- у пользователя могут быть активные задачи с `due_to = null`.

**Основной сценарий:**

1. Пользователь отправляет «без даты» или `/undated`.
2. Handler вызывает `TaskService.list_undated(user)`.
3. Сервис выбирает только задачи текущего пользователя: `due_to IS NULL`, `status = active`.
4. Бот отправляет список задач с inline-действиями «Назначить дату» и «Удалить».
5. Если задач много, список ограничивается страницей до 5 задач.

**Постусловия:**

- пользователь видит только свои активные задачи без даты;
- выбор действия ведёт в UC-06 или UC-07.

**Исключения:**

- Задач без даты нет: бот сообщает об этом.
- Устаревшая callback-кнопка пересчитывает актуальное состояние и не меняет чужие или неактивные задачи.

**Связанные компоненты:** `UndatedHandler`, `TaskService`, `TelegramSender`.

---

### UC-06. Назначение или редактирование даты задачи

**Цель:** пользователь уточняет дату задачи после создания или меняет уже назначенную дату.

**Предусловия:**

- задача принадлежит текущему пользователю;
- задача активна;
- задача доступна из списка, карточки, напоминания или вечернего диалога.

**Основной сценарий:**

1. Пользователь выбирает «Назначить дату» или «Изменить дату».
2. Бот просит ввести дату и время свободным русским текстом.
3. Пользователь отправляет дату.
4. `ParserService.parse_date()` нормализует дату в `Europe/Moscow`.
5. `TaskService` проверяет принадлежность и активность задачи.
6. Сервис обновляет `Task.due_to`.
7. Старые будущие reminders отменяются, заменяются или исключаются из обработки.
8. Если дата содержит будущее точное время, создаётся новый актуальный `Reminder`.
9. Система создаёт `TaskEvent(date_set)` или `TaskEvent(rescheduled)`.
10. Бот подтверждает изменение.

**Постусловия:**

- задача с назначенной датой исчезает из списка «без даты»;
- будущие напоминания соответствуют новой дате;
- старые будущие reminders не отправятся.

**Исключения:**

- Дата не распознана: бот просит повторить ввод.
- Дата в прошлом: изменение отклоняется.
- Задача уже `done`, `cancelled` или `deleted`: действие недоступно.
- Задача не принадлежит пользователю: действие отклоняется.

**Бизнес-правила:**

- Если пользователь указал только дату без времени, задача может попасть в список дня, но точечный `Reminder` не создаётся до уточнения времени.
- Состояние ожидания даты должно храниться в PostgreSQL или быть восстановимым; его нельзя держать только в памяти worker/handler.

**Связанные компоненты:** `TextHandler`, `CallbackHandler`, `ParserService`, `TaskService`, `ReminderService`.

### UC-07. Удаление задачи через кнопку

**Цель:** пользователь убирает ненужную задачу из активных списков и будущих напоминаний.

**Предусловия:**

- задача принадлежит текущему пользователю;
- задача активна;
- задача показана в списке, карточке или сообщении бота.

**Основной сценарий:**

1. Пользователь нажимает inline-кнопку «Удалить».
2. `TaskService` проверяет владельца и статус задачи.
3. Сервис переводит задачу в `deleted`.
4. Будущие reminders по задаче отменяются или исключаются из обработки.
5. Система создаёт `TaskEvent(deleted)`.
6. Бот подтверждает удаление и обновляет список.

**Постусловия:**

- задача не попадает в активные списки, дайджесты и due reminders;
- запись остаётся в БД для истории.

**Исключения:**

- Задача не найдена или уже неактивна: бот сообщает, что действие недоступно.
- Повторное нажатие старой кнопки не создаёт повторное событие удаления.

**Связанные компоненты:** `CallbackHandler`, `TaskService`, `ReminderService`, `TaskEvent`.

### UC-08. Утренний дайджест в 09:00

**Цель:** пользователь каждое утро получает список задач на день.

**Предусловия:**

- Celery Beat, Redis broker и Celery Worker запущены;
- у пользователя могут быть активные задачи на текущий день;
- расчёт дня выполняется в `DEFAULT_TIMEZONE` (`Europe/Moscow` по умолчанию).

**Основной сценарий:**

1. Celery Beat в `MORNING_DIGEST_HOUR`:00 MSK (по умолчанию 09:00) публикует job утреннего дайджеста.
2. Worker вызывает `MailingService`.
3. `MailingService` выбирает активные задачи на сегодня, сгруппированные по пользователям.
4. Для каждого пользователя `TelegramSender` отправляет короткий список задач.
5. Система фиксирует факт отправки через `TaskEvent(digest_sent)` или другую delivery-запись, если выбран такой дизайн.

**Постусловия:**

- пользователь видит только свои задачи;
- пустой дайджест не отправляется;
- повторный запуск за тот же день не отправляет дубль.

**Исключения:**

- Telegram timeout/5xx: ограниченный retry.
- Telegram 429: учитывать `retry_after` и снизить темп.
- Пользователь заблокировал бота: не ретраить бесконечно, зафиксировать постоянную ошибку.
- БД недоступна: не отправлять сообщение, если результат нельзя зафиксировать.

**Бизнес-правила:**

- Celery Beat только публикует job, но не выбирает пользователей и не вызывает Telegram напрямую.
- Идемпотентность дайджеста должна опираться на PostgreSQL, например ключ `user + date + mailing_type`.
- Реакция на дайджест не меняет статус задачи, если отдельный сценарий не реализован.

**Связанные компоненты/пайплайны:** Celery Beat, Redis, Celery Worker, `MailingService`, `TaskService`, `TelegramSender`, PostgreSQL.

### UC-09. Напоминание в назначенное время

**Цель:** бот напоминает пользователю о задаче в запланированное время.

**Предусловия:**

- есть `Reminder` с наступившим или прошедшим `reminder_time`;
- связанная задача активна;
- `Reminder.sent_time` ещё не заполнен.

**Основной сценарий:**

1. Celery Beat каждые `REMINDER_CHECK_INTERVAL_MINUTES` мин (по умолчанию 1) публикует job due reminders.
2. Worker вызывает `ReminderService`/`MailingService`.
3. Сервис claim-ит ограниченную пачку due reminders в PostgreSQL.
4. Перед отправкой система повторно проверяет, что задача активна.
5. `TelegramSender` отправляет напоминание пользователю.
6. Система сохраняет `sent_time` и `message_id`.
7. Система создаёт `TaskEvent(reminder_sent)`.
8. Пользователь может поставить реакцию ✅ или ❌ под этим сообщением.

**Постусловия:**

- напоминание отправлено не более одного раза;
- reaction handler может найти задачу по `chat_id + message_id`;
- просроченное, но не обработанное напоминание можно доставить позже после рестарта worker.

**Исключения:**

- Задача стала `done`, `cancelled` или `deleted` между claim и отправкой: сообщение не отправляется.
- Telegram API временно недоступен: ограниченный retry и диагностируемый статус.
- Worker упал: следующий запуск восстанавливает состояние из PostgreSQL.
- Одна и та же job попала в очередь дважды: второй запуск видит уже отправленное/обрабатываемое напоминание.

**Бизнес-правила:**

- Источник отправки - `Reminder`, а не прямое сканирование всех задач по `Task.due_to`.
- Обработка идёт пачками, чтобы уважать rate limit.
- Поминутный job не гарантирует точность до секунды; небольшое опоздание допустимо для MVP.

**Связанные компоненты/пайплайны:** Celery Beat, Redis, Celery Worker, `ReminderService`, `MailingService`, `TelegramSender`, PostgreSQL.

---

### UC-10. Отметка задачи выполненной

**Цель:** пользователь сообщает, что задача выполнена.

**Предусловия:**

- бот ранее отправил напоминание;
- `Reminder.message_id` сохранён;
- пользователь поставил реакцию ✅.

**Основной сценарий:**

1. Telegram присылает update о реакции ✅.
2. `ReactionHandler` достаёт `chat_id` и `message_id`.
3. `TaskService.mark_done_by_reminder()` находит `Reminder` по `chat_id + message_id`.
4. Система проверяет, что финальная реакция ещё не сохранена.
5. `Reminder.reaction` сохраняется как ✅/done.
6. `Task.status` меняется на `done`.
7. Будущие reminders по задаче отменяются или исключаются.
8. Создаётся `TaskEvent(completed)`.
9. Бот может подтвердить выполнение.

**Постусловия:**

- задача больше не попадает в активные списки, due reminders и вечерний перенос;
- первая финальная реакция зафиксирована в истории.

**Исключения:**

- Неизвестный `message_id`: игнорировать или логировать.
- Повторная или изменённая реакция: не менять финальный статус.
- Задача уже отменена/удалена: не возвращать её в активное состояние.

**Связанные компоненты:** `ReactionHandler`, `TaskService`, `ReminderService`, `TaskEvent`.

---

### UC-11. Отмена задачи реакцией

**Цель:** пользователь отменяет задачу, и бот больше не напоминает о ней.

**Предусловия:**

- бот ранее отправил напоминание;
- `Reminder.message_id` сохранён;
- пользователь поставил реакцию ❌.

**Основной сценарий:**

1. Telegram присылает update о реакции ❌.
2. `ReactionHandler` достаёт `chat_id` и `message_id`.
3. `TaskService.cancel_by_reminder()` находит связанный `Reminder`.
4. Система сохраняет реакцию.
5. `Task.status` меняется на `cancelled`.
6. Будущие reminders по задаче отменяются или исключаются.
7. Создаётся `TaskEvent(cancelled)`.
8. Бот может подтвердить отмену.

**Постусловия:**

- задача не попадает в активные списки, дайджесты, due reminders и вечерний перенос.

**Исключения:**

- Реакция пришла на неизвестное или устаревшее сообщение: игнорировать или логировать.
- Задача уже `done`, `cancelled` или `deleted`: повторное изменение не выполняется.

**Бизнес-правила:**

- ❌ означает окончательную отмену задачи в MVP.
- Удаление через кнопку использует статус `deleted`, но даёт похожий пользовательский эффект.

**Связанные компоненты:** `ReactionHandler`, `TaskService`, `ReminderService`, `TaskEvent`.

---

### UC-12. Вечерняя проверка и перенос

**Цель:** если пользователь не отреагировал на напоминание, бот вечером предлагает перенести задачу.

**Предусловия:**

- наступило `EVENING_MISSED_CHECK_HOUR`:00 MSK (по умолчанию 20:00);
- есть активная задача с отправленным `Reminder`;
- у reminder нет финальной реакции.

**Основной сценарий:**

1. Celery Beat в `EVENING_MISSED_CHECK_HOUR`:00 MSK (по умолчанию 20:00) публикует job вечерней проверки.
2. Worker вызывает `MailingService.ask_about_missed_reminders(now)`.
3. Сервис выбирает отправленные reminders без реакции и с активными задачами.
4. Бот спрашивает пользователя, на какую дату и время перенести задачу.
5. Пользователь отвечает свободным русским текстом.
6. `ParserService.parse_date()` нормализует дату.
7. `TaskService.reschedule()` обновляет `Task.due_to`.
8. Старые будущие reminders отменяются или заменяются.
9. Создаётся новое актуальное напоминание, если есть будущие дата и точное время.
10. Создаётся `TaskEvent(rescheduled)`.
11. Бот подтверждает перенос.

**Постусловия:**

- задача остаётся активной, но получает новую дату;
- старые неактуальные reminders больше не отправляются;
- повторный вечерний job не задаёт один и тот же вопрос несколько раз.

**Исключения:**

- Переносить нечего: бот ничего не отправляет.
- Пользователь не отвечает: для MVP задача остаётся активной, если команда не зафиксирует другое правило.
- Дата не распознана: бот просит повторить ввод.
- Дата в прошлом: бот просит будущую дату.
- Пользователь отвечает ❌ вместо даты: действие можно трактовать как отмену только если это явно реализовано в handler.

**Бизнес-правила:**

- Вечерний вопрос должен иметь идемпотентность, например по `task/reminder + date + mailing_type`.
- `message_id` вечернего вопроса сохраняется, если по нему ожидается продолжение диалога.
- Состояние ожидания ответа должно переживать рестарт процесса.

**Связанные компоненты/пайплайны:** Celery Beat, Redis, Celery Worker, `MailingService`, `TaskService`, `ReminderService`, `ParserService`, `TelegramSender`.

## Follow-up Use Cases

### UC-F01. Полный цикл повторяющихся задач

**Цель:** после выполнения или наступления очередного периода система планирует следующее напоминание по `repeat_type` и `repeat_interval`.

**Источник:** пользователь может назвать интервал повторения в `tz/tz.md`; parser уже сохраняет поля повторения в MVP.

**Основной сценарий:**

1. Пользователь создаёт задачу с повторением.
2. Система сохраняет `repeat_type` и `repeat_interval`.
3. После выполнения текущего occurrence или по отдельному scheduled job система рассчитывает следующий occurrence.
4. Создаётся новый актуальный `Reminder` или новая задача-экземпляр, в зависимости от выбранной модели.
5. Система пишет `TaskEvent`.

**Бизнес-правила:**

- MVP parser допускает `daily`, `weekly`, `monthly`.
- Сложные RRULE, исключения календаря и перенос отдельных occurrence остаются post-MVP.

### UC-F02. Статистика за неделю

**Цель:** пользователь видит краткую статистику по своим задачам.

**Источник:** усложнение из `tz/tz.md`.

**Основной сценарий:**

1. Пользователь отправляет `/stats`.
2. Система выбирает `TaskEvent` текущего пользователя за последние 7 дней.
3. Система считает выполненные вовремя, перенесённые, отменённые и удалённые задачи.
4. Бот отправляет краткий отчёт.

**Бизнес-правила:**

- Источник статистики - PostgreSQL, прежде всего `TaskEvent`.
- События должны быть идемпотентными: повторная реакция не создаёт дубль статистики.

---

### UC-F03. Расширенная пагинация списков

**Цель:** длинные списки задач остаются удобными в Telegram.

**Основной сценарий:**

1. Сервис списка получает пользователя, фильтр, номер страницы и размер страницы.
2. Система возвращает максимум 5 активных задач.
3. Бот показывает задачи и inline-кнопки навигации.

**Бизнес-правила:**

- Базовое ограничение страницы полезно для MVP, полноценная навигация может быть follow-up.
- Пагинация должна жить в сервисном слое, а не в handler.

## Сквозные бизнес-правила

- Все операции выполняются строго per user по `chat_id`.
- Активные списки показывают только задачи со статусом `active`.
- Задачи `done`, `cancelled` и `deleted` не получают будущие напоминания.
- PostgreSQL является единственным источником истины для бизнес-состояния.
- Redis не хранит задачи, реакции, `message_id`, историю доставок или состояние диалога как единственную копию.
- Все даты нормализуются и сравниваются в `Europe/Moscow`.
- Дату в прошлом нельзя назначать или сохранять при создании.
- Задача без даты не получает точечный `Reminder`.
- Задача с датой без точного времени может попасть в список дня, но не получает точечный `Reminder` до уточнения времени.
- `Task`, `Reminder` и `TaskEvent` при создании задачи пишутся в одной транзакции.
- Telegram handlers и Celery tasks должны быть тонкими.
- Внешние API-клиенты должны быть заменяемыми mock-объектами в тестах.
- Секреты, токены, API keys и полные приватные тексты задач не должны попадать в логи и ответы пользователю.
- Логи и пользовательские сообщения в проекте пишутся на русском.

## Минимальные тесты

- `UserService` создаёт пользователя по `chat_id` и не создаёт дубли.
- Voice handler отклоняет длинное/большое voice до вызова SpeechKit.
- Voice handler при успешном flow вызывает SpeechKit, YandexGPT и `TaskCreationService`.
- Text handler вызывает YandexGPT и `TaskCreationService` без STT и скачивания файла.
- Parser валидирует JSON Schema, пустой `title`, `due_to` в прошлом и повторения.
- `TaskCreationService` создаёт `Task`, создаёт `Reminder` только при будущей точной дате и всегда создаёт `TaskEvent`.
- При ошибке SpeechKit, GPT, parser или БД задача не создаётся частично.
- `/undated` показывает только активные задачи текущего пользователя без даты.
- Назначение даты обновляет `Task.due_to` и заменяет будущие reminders.
- Удаление переводит задачу в `deleted` и исключает будущие reminders.
- Утренний дайджест фильтрует по пользователю, не отправляет пустой список и не дублируется.
- Due reminders отправляются один раз, сохраняют `sent_time` и `message_id`.
- Реакции ✅/❌ применяются только один раз и отменяют будущие reminders.
- Вечерняя проверка выбирает только отправленные reminders без реакции.
- Retry policy различает временные ошибки и постоянные ошибки Telegram/Yandex.
- Идемпотентность сохраняется при повторном запуске scheduled job.
