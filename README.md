# Голосовой напоминальщик

Telegram-бот для проекта стажировки Иви-2026: пользователь быстро создаёт задачи голосом или текстом, получает напоминания, отмечает результат реакциями и переносит невыполненные дела.

Проблема: иногда нужно быстро записать дело, чтобы не забыть, но вручную заводить задачу неудобно. Целевая аудитория — занятые люди, которым проще сказать боту «напомни завтра в 15:00 позвонить врачу», чем открывать отдельное приложение.

Основная ценность MVP — Telegram-бот, который принимает голос или текст, превращает свободную русскую фразу в задачу, хранит её в PostgreSQL и сам напоминает в нужное время.

## Быстрый старт (Docker)

Требуются Docker Desktop с Compose v2 и `make`.

```bash
cp .env.example .env
# Заполните DJANGO_SECRET_KEY и TELEGRAM_BOT_TOKEN в .env.
make up
make ps
```

`make up` собирает один образ приложения и запускает PostgreSQL, Redis, миграции,
Telegram-бота, Celery worker и Celery Beat. Контейнер `migrate` должен завершиться
с кодом `0`, остальные сервисы должны иметь состояние `Up`/`healthy`.

```bash
make logs                         # логи bot, worker и beat
docker compose --profile app logs migrate  # результат применения миграций
make down                         # остановить стек, сохранив данные PostgreSQL
```

Для голосового сценария дополнительно заполните `YANDEX_API_KEY` и
`YANDEX_FOLDER_ID`. При `PARSER_BACKEND=mock` разбор задачи не требует вызова
YandexGPT, но распознавание voice по-прежнему использует SpeechKit.

## Разработка с pipenv на хосте

Старый workflow сохранён: приложение работает на хосте, а в Docker запускаются
только PostgreSQL и Redis. Значения `POSTGRES_HOST=localhost` и
`REDIS_URL=redis://localhost:6379/0` из `.env.example` предназначены именно для
этого режима.

```bash
make install
make up-infra
make migrate
pipenv run python manage.py runbot

# В отдельных терминалах:
make worker
make beat
```

Не запускайте Celery Beat одновременно на хосте и в Docker: два планировщика
могут поставить одну и ту же job в очередь дважды.

Проверки проекта по-прежнему выполняются на хосте и не требуют контейнеризации:

```bash
make check
make lint
make test
```

## Docker: режимы и диагностика

| Команда | Результат |
|---|---|
| `make up` | Полный MVP-стек в фоне с пересборкой образа |
| `make up-infra` | Только PostgreSQL и Redis для приложения на хосте |
| `make ps` | Состояние всех сервисов, включая one-shot `migrate` |
| `make logs` | Поток логов bot, worker и beat |
| `make down` | Остановка полного стека без удаления volumes |
| `make migrate-docker` | Ручной запуск миграций в контейнере |

Внутри Compose `POSTGRES_HOST` переопределяется на `postgres`, а `REDIS_URL` —
на `redis://redis:6379/0`. Значения `localhost` внутри app-контейнеров не
работают, потому что там они указывают на сам контейнер.

| Симптом | Что проверить |
|---|---|
| bot постоянно перезапускается | `docker compose --profile app logs migrate` и `docker compose --profile app logs bot`; наличие корректного токена в `.env` |
| напоминания не приходят | `make ps`, затем `docker compose --profile app logs worker` и `docker compose --profile app logs beat` |
| `connection refused` к PostgreSQL | внутри контейнера `POSTGRES_HOST` должен быть `postgres`, не `localhost` |
| после `git pull` запускается старый код | повторить `make up` или `docker compose --profile app up -d --build` |

Для полного удаления локальных данных используйте
`docker compose --profile app down -v`; обычный `make down` volumes сохраняет.

## Как прогнать E2E

Сценарий соответствует документу [`tz/use_cases.md`](tz/use_cases.md) и отчёту TG-10.

1. Создать `.env`, заполнить Telegram-токен и Yandex credentials для voice.

```bash
make up
```

2. Убедиться, что миграции завершились, а процессы запущены:

```bash
make ps
docker compose --profile app logs migrate
```

3. Отправить боту `/start`, затем голосовое сообщение:

```text
Напомни завтра в 15:00 позвонить врачу
```

4. Проверить:

- задача создана в БД;
- создано напоминание;
- в назначенное время бот отправил сообщение;
- кнопки `Сделано` и `Удалить` изменяют состояние задачи.

5. Перезапустить worker и убедиться, что он снова подключился к Redis:

```bash
docker compose --profile app restart worker
docker compose --profile app logs worker
```

Подробные пользовательские сценарии приведены в `tz/use_cases.md`.

## Процесс разработки

Команда ведёт работу через **Kanban-доску GitHub Projects** репозитория и pull request'ы в `main`. Полный список задач, оценок и зависимостей — в `[decomp.md](decomp.md)`; тексты тикетов — в каталоге `[issues/](issues/)`.

### Kanban-доска

Доска отражает жизненный цикл одной задачи. Перемещай карточку только когда выполнено действие из колонки «Когда перемещать».


| Колонка         | Смысл                                                    | WIP limit | Когда перемещать                                                                                       |
| --------------- | -------------------------------------------------------- | --------- | ------------------------------------------------------------------------------------------------------ |
| **Backlog**     | Задача заведена, но работа ещё не начата                 | —         | Карточка попадает сюда при создании issue и добавлении на доску. Перед взятием в работу — в **Ready**. |
| **Ready**       | Задача готова к взятию: зависимости закрыты, есть оценка | —         | Берёшь задачу — сразу в **In progress**. Не держи карточку здесь, если уже начал кодить.               |
| **In progress** | Задача в активной разработке                             | 3         | Открыл PR — в **In review**. Если работа остановлена — верни в **Ready** или **Backlog**.              |
| **In review**   | Код на ревью, PR открыт                                  | —         | PR смержен — в **Done**. Если нужны правки — обратно в **In progress**.                                |
| **Done**        | Задача завершена                                         | —         | Финальное состояние после merge PR.                                                                    |


Поток по умолчанию:

```text
Backlog → Ready → In progress → In review → Done
```



#### WIP limits

WIP limit — максимум карточек в колонке одновременно. На доске действует один лимит:

- **In progress** — 3

Если лимит **In progress** исчерпан, сначала доведи до конца или верни в предыдущую колонку одну из текущих задач, и только потом бери новую. Это снижает переключение контекста и помогает ментору видеть реальную загрузку команды.

#### Оценки (Estimate)

У каждой карточки на доске заполняется поле **Estimate** — трудозатраты в часах, как в `[decomp.md](decomp.md)` (`2ч`, `3ч`, `4ч` и т.д.). В заголовке каждой колонки доска показывает **сумму Estimate** по карточкам в ней — так видно, сколько работы «лежит» в backlog, уже в review и т.д.

Правила:

- Оценку проставляет тот, кто берёт задачу, сверяясь с таблицей в `decomp.md`. Если фактический объём сильно отличается — обнови Estimate и коротко зафиксируй причину в issue.
- Не начинай работу без оценки: карточка без Estimate не должна попадать в **Ready**.
- Одна карточка = один тикет (`APP-01`, `TG-04`, `BG-QA-02` и т.п.). Не объединяй несколько тикетов в одну карточку.



#### Связь задач с доской

1. Создай **GitHub Issue** с заголовком в формате `<ID>: <название>`, например `APP-01: Создать Django-проект и приложение reminder`. Текст можно взять из `[issues/<эпик>/<ID>.md](issues/)`.
2. Добавь issue на Kanban-доску проекта — карточка появится в **Backlog**.
3. Заполни **Estimate** и проверь зависимости из `decomp.md` — затем перенеси в **Ready**.
4. В описании issue укажи эпик-префикс и ссылку на markdown-файл тикета, если он есть.

Префиксы эпиков: `APP`, `DB`, `CORE`, `TG`, `AI`, `BG`, `QA`, `DOC`.

### Pull request'ы



#### Именование веток

В репозитории пока нет зафиксированной истории веток; используем соглашение по ID тикета:

```text
<ID>-<краткое-описание-латиницей>
```

Примеры:

- `APP-01-django-project`
- `TG-04-text-task-creation`
- `DB-02-task-model`

Ветку создавай от актуального `main`. Одна ветка — один тикет.

#### Заголовок PR

```text
[<ID>] <Краткое описание на русском>
```

Пример: `[APP-01] Создать Django-проект и приложение reminder`

#### Описание PR

Используй шаблон:

```markdown
## Связанная задача

Closes #<номер-issue>

## Что сделано

- ...

## Как устроено

- Ключевые файлы и модули — с чего начать чтение кода
- Кратко: поток данных, основные классы/функции, связи между компонентами
- Неочевидные решения, которые стоит знать при ревью

## Заметки для ревьюера

- На что обратить внимание в первую очередь
- Что сознательно не делали / отложили на потом
- Вопросы, по которым нужен совет
```

- `Closes #<номер>` (или `Fixes #<номер>`) — обязательная связь с GitHub Issue. После merge issue закроется автоматически.
- Блоки «Как устроено» — главная часть описания: помогают ментору быстро понять мотивацию, архитектурные решения и где искать код. Пиши конкретно, со ссылками на файлы.
- Блок «Как проверить» — **опциональный**. Добавляй только если без ручного шага неочевидно, что всё работает (например, нестандартная команда запуска или граничный сценарий). Не дублируй полный чеклист из тикета — ментор не прогоняет exhaustive QA по каждому PR.
- Блок «Заметки для ревьюера» — фокус ревью: риски, спорные места, что проверить в коде, а не пошаговый manual test.
- Если PR закрывает только часть issue — используй `Refs #<номер>` и опиши, что осталось.



#### Движение карточки и PR


| Действие                             | Колонка на доске          |
| ------------------------------------ | ------------------------- |
| Начал работу над задачей             | **In progress**           |
| Открыл PR                            | **In review**             |
| PR смержен в `main`                  | **Done**                  |
| PR закрыт без merge, работа отменена | **Backlog** или **Ready** |


PR не должен висеть в **In review**, пока карточка ещё в **In progress**, и наоборот.
