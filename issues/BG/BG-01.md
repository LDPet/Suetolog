# BG-01: Celery + Beat + `send_due_reminders` (merged BG-01/02/03)

Тег: BG-01

Эпик: EPIC-05. Celery, Redis и фоновые рассылки

Зависимости: APP-01, APP-04, CORE-05, TG-01

## Зачем этот тикет

**Первый вертикальный срез фона.** Поднимаем Celery/Redis/Beat и сразу подключаем первый рабочий job — точечные напоминания. После закрытия BG-01 можно тестировать реакции (TG-08) и продолжать дайджест/вечер (BG-05/06).

> Бывшие BG-02 (расписание Beat) и BG-03 (`send_due_reminders`) — **не отдельные тикеты**.

## Технологии и контекст

```
Celery Beat → Redis broker → Celery worker → ReminderService / MailingService → TelegramSender → PostgreSQL
```

- **PostgreSQL** — источник правды: кого слать, `sent_time`, `message_id`, `TaskEvent`.
- **Redis** — только broker; бизнес-состояние не хранить в Redis.
- **Celery task** — тонкая обёртка: вызывает сервис рассылки, не дублирует бизнес-логику CORE-05.
- **Beat** — планировщик; в этом тикете регистрируем расписание **due reminders** (~каждую минуту). Слоты digest/evening — placeholder или комментарий «реализуется в BG-05/06».
- Документация: `tz/MAILING_PIPELINES.md`, `tz/ARCHITECTURE.md`.

## Границы тикета

**Входит:** `config/celery.py`, worker + beat локально, settings для интервалов, task `send_due_reminders`, регистрация в Beat, базовая защита от повтора по `sent_time`.

**Не входит:** утренний дайджест (BG-05), вечерняя проверка (BG-06), claim/lock и параллельные workers (BG-F01), retry policy 429/5xx (BG-F02), автотесты фона (BG-F03), обработка реакций (TG-08).

## Правило: `TaskEvent` на отправку reminder

| Действие | `event_type` | `message_id` |
| --- | --- | --- |
| Отправлено напоминание по due time | `reminder_sent` | да |

Событие пишет **сервисный слой** (CORE-05 / mailing service), Celery task только оркестрирует.

## Что реализуется

1. **`config/celery.py`** — Celery app, autodiscover tasks, Beat schedule из settings.
2. **Settings** — минимум: `REMINDER_CHECK_INTERVAL_MINUTES` (default ~1), `DEFAULT_TIMEZONE` (`Europe/Moscow`). Константы digest/evening можно объявить заранее без реализации jobs.
3. **Task `send_due_reminders`** (имя модуля на усмотрение):
   - вызов `ReminderService` / mailing service: выбор due active reminders **пачкой** (лимит batch);
   - для каждого кандидата: **пропустить**, если `sent_time` уже установлен;
   - `TelegramSender` → сохранить `sent_time`, `message_id` на Reminder;
   - `TaskEvent(reminder_sent, message_id=...)`.
4. **Beat schedule** — job due reminders по `REMINDER_CHECK_INTERVAL_MINUTES`.
5. **Makefile / README** — как запустить worker и beat локально (рядом с `runbot`).

## Кривые кейсы — учесть

| Ситуация | Ожидание MVP |
| --- | --- |
| Worker не запущен, due time прошёл | После старта worker следующий beat-run догоняет просроченные (если `sent_time` пуст) |
| Telegram send упал до записи в БД | Лог ошибки; повторный run может отправить снова — **known limitation** до BG-F01 |
| Несколько reminders одному user | Отправка пачкой, без падения job на первой ошибке (остальные обработать или залогировать) |
| Опоздание на 1–2 мин относительно due time | Допустимо для MVP (интервал beat) |

## Acceptance criteria

- Worker и beat стартуют локально с Redis из APP-04
- Beat запускает `send_due_reminders` по расписанию из settings
- Due reminder: Telegram-сообщение + `sent_time` + `message_id` + `TaskEvent(reminder_sent)`
- Повтор job не шлёт reminder с уже заполненным `sent_time`
- Celery task не содержит ORM-логики выбора кандидатов — только вызов сервиса
- Секреты не в логах
